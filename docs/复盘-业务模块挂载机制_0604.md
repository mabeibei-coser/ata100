# ATA100 复盘：业务模块怎么挂进来 + 12 个坑

> 用于：ATA100（薪酬域）复用本套形态时，照抄什么、改什么、避什么。
> 复盘时点：2026-06-04
> ATA100 当前形态：会员中心 + A600 薪资查询 + A800 岗位全景文档 三产品在 `h100.jsai100.com` 同域三子路径

---

## 一、ATA100 在干什么（一句话）

**它是个"登录 + 付费 + 身份"底座**，本身不做业务功能。A600 / A800 是独立产品（独立仓库、独立端口、独立 DB），通过**共享 cookie + 只读挂载 DB** 接进来。

把 ATA100 想成 C 端的 admin-hub —— admin-hub 是后台聚合各业务全量数据，ATA100 是 C 端聚合每个用户自己的跨产品历史。

## 二、业务模块"接进来"的 5 个机制（这是 skill 核心）

### 机制 1：共享 Cookie（登录态跨产品复用）

- **谁签发**：只有 ATA100 签发；A600/A800 只读解析 cookie 拿 phone，不自己签
- **cookie 名**：`ata_member_session`（domain-level constant）
- **密钥**：`ATA_MEMBER_SESSION_PASSWORD`，三产品 env 必须**填同一个值**（48 字节随机串）
- **path 设置**：`/`（全站可见）；session 只装 `{phone, userId}` 几十字节，全站广播代价可忽略
- **域分隔机制（关键！）**：薪酬域将来用**另一个 cookieName + 另一个密钥**，浏览器里两域 cookie 并存、互不解密 → VIP 天然不互通
- **实现库**：`iron-session`（带签名+加密的 httpOnly cookie，浏览器端拿不到明文）

### 机制 2：只读挂载业务 DB（个人中心聚合）

- **聚合谁的什么**：A600 的 `reports` 表（识别历史）+ A800 的 `document_downloads` 表（下载历史），按 phone 过滤
- **怎么挂**：`better-sqlite3` 用 `{readonly: true, fileMustExist: true}` 打开业务 DB 文件
- **DB 路径配在哪**：env 变量 `ATA_HAZARD_DB_PATH` / `ATA_DOC_DB_PATH`，指向业务产品 `data/*.db` 的绝对路径
- **3 条铁律**（写在 `lib/history.js` 注释里）：
  1. **只 SELECT，绝不 UPDATE/INSERT/DELETE 别人的库**
  2. **用独立连接**（不复用会员主连接 `getDb()`），避免 ATTACH 污染主连接
  3. **必须校验 `user_phone === 当前 phone`**（防越权看别人的数据）
- **降级策略**：业务库可能不存在（产品没跑过/未部署）→ readonly 打开会抛 → catch 后**返空数组**，不报错（个人中心展示"暂无"比白屏好）

### 机制 3：VIP 全域通用（时间制 + 幂等）

- **本质**：`memberships.vip_expire_at > now` 即为 VIP；读时比对，无需定时任务
- **写权**：只有 ATA100 写（微信支付回调里）
- **读权**：A600/A800 通过 `GET /api/membership/me` 向 ATA100 "刷卡" 校验 VIP，不自己判
- **幂等命门**：`membership_ledger.related_order_id` 加 UNIQUE 索引 → 微信回调重复推不会重复开通
- **续费叠加**：未过期续费往后叠（`base = max(到期, now)`），已过期从 now 起

### 机制 4：跨产品回跳（OAuth 风格的 from 参数 + 白名单）

- **场景**：用户在 A600 未登录 → 点登录 → 跳 ATA100 → 登完跳回 A600（不是停在中心首页）
- **实现**：URL 带 `?from=https://.../a600/`；登录完用 `resolveRedirect(from)` 验证 host 是否在白名单 → 通过则 redirect
- **白名单 env**：`ATA_ALLOWED_HOST=h100.jsai100.com`（不配则只允许站内相对路径，业务模块跳不回）
- **辅助函数**：`isSafeFromPath()` + `resolveRedirect()` 在 `lib/wechat-oauth.js`

### 机制 5：跨域天然隔离（VIP / cookie / DB 全部不共享）

- **设计哲学**（已写在 `lib/session.js` 注释里）：薪酬域用另一个 cookieName + 另一个密钥 → 两域 cookie 浏览器并存、**互不解密**
- **VIP 是单域内通用、跨域独立**：
  - **单域内**：用户在 A600 开通 VIP → 同域的 A800 也认（共享 `memberships` 表）
  - **跨域**：用户在 ASG 开通 VIP → ATA 不认（两个 `memberships` 表在不同 DB）
- **常见误解**："两个域都是我的产品、用户应该一个 VIP 通用"。**反直觉但对**——业务上 ASG（薪酬）和 ATA（薪酬）就是两套产品、两套套餐价、两套权益，VIP 互不抵扣才符合业务直觉
- **三种用户场景全部 OK**：
  - 只用 ASG 的人 → 开 ASG VIP 就行
  - 只用 ATA 的人 → 开 ATA VIP 就行
  - 两边都用的人 → 分别开两个 VIP，互不抵扣
- **admin-hub 视角不受影响**：admin-hub 仍能跨域读两份 `memberships` 表做全量分析（同一 phone 在哪些域开了 VIP / 总付费多少），但前台分发各走各的
- **如果哪天业务变成"受众有交叉、想跨域 VIP 互通"**：要单独加一层"跨域 VIP 表"，目前不支持也不打算支持

## 三、12 个挂载坑（按踩坑频率排）

| # | 坑 | 症状 | 解药 |
|---|---|---|---|
| 1 | **cookieName / password / path 任一不一致** | 业务模块 cookie 解不开、所有用户都"未登录" | 三产品 env 同一份、cookieOptions 逐字节同 |
| 2 | **`ATA_COOKIE_PATH` 设了子路径** | 业务模块在不同子路径读不到 cookie | 必须 `/`，不能收敛到 `/ata100` |
| 3 | **VITE_BASE_PATH 是编译时变量** | build 后子路径请求 404 | 每个产品 build 前必须 export 对自己的子路径 |
| 4 | **业务 DB 文件不存在** | history 接口 500 | `fs.existsSync` 先查 + `try/catch` 兜底返空 |
| 5 | **业务 DB 用主连接 ATTACH** | 污染会员主连接、后续操作异常 | 用**独立 readonly 连接**（`new Database(path, {readonly:true})`），用完 close |
| 6 | **没校验 user_phone 越权** | 用户 A 看到用户 B 的报告 | 每个 history 查询都加 `WHERE user_phone = ?` 用当前 session.phone |
| 7 | **微信回调验签用 JSON 解析后 body** | 验签失败、VIP 开不出来 | `/api/pay/wechat/notify` 路由**跳过全局 json middleware**，用原始 bytes 验签 |
| 8 | **VIP 入账没做幂等** | 微信重推回调 → VIP 被开两次、流水重复 | `membership_ledger.related_order_id` UNIQUE 索引 + 入账前先查 |
| 9 | **业务模块上线但中心没上** | 老用户点登录跳"登不进的中心" → 砸生产 | **顺序铁律：中心先起 → 短信验证通 → 再放业务模块** |
| 10 | **跨产品回跳 from 参数被注入 XSS** | 钓鱼跳第三方站 | `ATA_ALLOWED_HOST` 白名单 + `resolveRedirect()` 验 host |
| 11 | **MASTER_OTP_CODE 上线没删** | 任何人输 `888888` 可登任何手机号 | 部署 checklist 写死 + 生产 env.example 注释里写"上线必删" |
| 12 | **短信宝签名/正文未报备** | 真实短信里签名被自动换成默认值（用户疑惑） | 复用已报备账户的签名 + 正文模板（与 A200 共用，一字不差） |

## 六、文档库复用 3 个方案 + 推荐

> **背景**：ATA100 要接"文档库"，而 A800 岗位全景文档已经在给 ATA100 用。
> **核心问题**：A800 怎么知道现在用户登的是 ASG 域还是 ATA 域？

### 方案 A：A800 同时认两套 cookie（minimal 改动）

- A800 的 session.js 同时定义 `ata_member_session` 和 `ata_member_session`，谁能解开用谁的 phone
- 文档库里的"我下载过的"按 phone 聚合，不区分来源域
- **优点**：A800 单实例、单 DB、文档维护一份
- **缺点**：① VIP 校验不知道找哪个中心（asg 还是 ata 的 `/api/membership/me`？）② 文档可见性如果按域区分会复杂

### 方案 B：A800 跑两份实例（强隔离）+ admin-hub 统一管理

- **C 端形态**：部署 `asg-doc-library`（4003）和 `ata-doc-library`（新端口），代码同源、DB 独立
- 每份实例只认一套 cookie + 一个会员中心
- **B 端形态（方案 B 的天然延伸）**：admin-hub 统一管理两份文档库
  - admin-hub 上传文档时选"目标域"（asg / ata）→ 直接写到对应实例的 DB + PDF 存到对应 `data/uploads/`
  - admin-hub 本来就是"全域写入 + 各产品只读自己 DB"的角色，多写一两个 DB 文件不破坏架构
  - 两个 doc-library 实例**代码同源**，迭代时改一份代码 → 重新部署两份实例（半小时活）
  - **默认"文档完全独立"**（asg 上传一份、ata 另一份），未来如要"一份文档分发到多个域"再加复选

```
                admin-hub (B100，统一上传)
                    │
       ┌────────────┴────────────┐
       │  上传时选"目标域"        │
       ▼                         ▼
 写 asg-doc 的 DB + 文件      写 ata-doc 的 DB + 文件
       │                         │
       ▼                         ▼
 asg-doc-library (4003)     ata-doc-library (新端口)
 只读自己 DB                  只读自己 DB
 暴露给 ATA100 前台           暴露给 ATA100 前台
```

- **优点**：C 端 cookie 隔离 + B 端统一管理，两边各自最优；blast radius 小（一边挂了不影响另一边）；薪酬文档和薪酬文档分域天然合理
- **缺点**：部署清单变长（多一个 pm2 进程 + 一段 nginx location）

### 方案 C：ATA100 自己内嵌一个 "doc-lite"，不复用 A800

- ATA100 仓库内自带一个简化版文档库（不抽独立产品）
- **优点**：架构最干净、ATA100 单仓库自洽
- **缺点**：A800 已经做好的 PDF 自动预览 / 文档分类 / 详情页等能力要重写或拷贝

### 我的推荐：**方案 B**（跑两份实例）

理由：
1. **域隔离是 ATA100 session.js 原注释里就写好的设计哲学**（"两域 cookie 并存、互不解密"）；A800 双 cookie 会破坏这个设计
2. **文档库的内容分域天然合理**：薪酬域的文档（应急预案、安全手册）和薪酬域的文档（薪酬制度、激励方案）受众完全不重，没有共享必要
3. **部署成本可控**：A800 代码已经成型，跑两份实例只是 nginx + pm2 + 改 env，半天搞定；ATA100 内嵌（方案 C）要重写一遍 UI/预览/分类，工作量大得多
4. **未来 ATA100 文档的访问控制可能和薪酬业务深度耦合**（比如"只 VIP 看奖金细则"），双实例可以独立演进

**但这是建议，最终方案由你定（用户/设计层决策）**。

## 七、ATA100 落地 step-by-step（基于 ATA100 形态）

> 不重发明轮子。下面是把 ATA100 立起来的最短路径。

1. **复制 ATA100 整个仓库** → 改名 `ATA100-薪酬域会员中心-ata100`（具体目录名待定）
2. **全局替换**：`ata100` → `ata100`、`ATA_` → `ATA_`、`ata_member_session` → `ata_member_session`、文案"薪酬" → "薪酬"
3. **重新生成 SESSION_PASSWORD**（48 字节随机）—— 必须新值，不能复用 ATA100 的
4. **改 `lib/history.js`**：把 hazard 聚合换成 A500 + 文档库（按方案 B：ata-doc-library）
5. **改 `lib/packages.js`**：按 ATA 业务的套餐（VIP 时长、价格、卖点）
6. **改 `src/`**：UI 文案 + 个人中心聚合卡片
7. **`server.js` 端口分配** → 新端口（4002/4003 已占，可走 4004+）
8. **部署清单 `deploy/DEPLOY.md`** → 按方案 B 改成"中心 + A500 + ata-doc-library 三产品一起上"
9. **跑 6 步必验**（登录 / cookie path / 跨产品认登录 / 跨产品回跳 / 真支付 / VIP 闭环）
